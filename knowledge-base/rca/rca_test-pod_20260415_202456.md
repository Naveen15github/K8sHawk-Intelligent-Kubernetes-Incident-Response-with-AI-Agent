## Root Cause Analysis
The pod is experiencing a CrashLoopBackOff due to an exit code 1. Analysis of pod logs shows the Java application is consuming excessive memory against a 2GB limit. The heap dump indicates a memory leak in the user session cache, where expired sessions are not being garbage collected properly. This is causing the container to exceed its memory limit and get terminated by the kubelet. The issue started after the recent deployment that increased concurrent user sessions without adjusting memory limits. This is impacting user authentication and session management, resulting in 5xx errors for approximately 30% of login attempts.
