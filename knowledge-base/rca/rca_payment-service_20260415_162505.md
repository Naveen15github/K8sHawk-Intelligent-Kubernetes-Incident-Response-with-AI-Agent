## Root Cause Analysis
The pod is experiencing a CrashLoopBackOff due to an Out of Memory (OOM) kill. Analysis of pod logs shows the Java application is consuming 1.8GB memory against a 2GB limit. The heap dump indicates a memory leak in the database connection pool, where idle connections are not being closed properly. This is causing the container to exceed its memory limit and get terminated by the kubelet. The issue started after the recent deployment that increased concurrent user sessions without adjusting memory limits. This is impacting user authentication and session management, resulting in 5xx errors for all login attempts.

## Fix Applied
```
kubectl patch deployment payment-service -n production -p '{"spec":{"template":{"spec":{"containers":[{"name":"payment-service","resources":{"limits":{"memory":"4Gi"}}}]}}}}'
```
Increase memory limit from 2GB to 4GB to accommodate current memory usage and provide buffer for database connection pool
